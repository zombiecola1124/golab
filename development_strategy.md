# 개발 전략 워크플로우 (Development Strategy)

golab 프로젝트의 모든 개발 작업은 아래 3단계 워크플로우를 따른다.

---

## 1단계: Plan (계획)

> 코드를 짜기 전에 항상 계획을 먼저 세우고, 사용자의 컨펌을 받는다.

- 새로운 기능이나 변경 사항이 있으면, 구현 전에 **계획서를 먼저 작성**한다
- 계획서에는 다음 내용을 포함한다:
  - 변경할 파일 목록
  - 구현 방법 요약
  - 예상되는 영향 범위 (다른 모듈에 미치는 영향)
  - 필요한 테스트 항목
- **사용자가 "진행해"라고 승인하기 전까지 코드를 작성하지 않는다**

## 2단계: Agent Team (다각도 검토)

> 편집장/개발자/검수자 페르소나를 가동해 다각도로 검토한다.

- **편집장 (Editor)**: 전체 아키텍처와 방향성이 올바른지 확인한다
  - 프로젝트 구조와 일관성 검토
  - CLAUDE.md 가이드라인 준수 여부 확인
  - 모듈 간 의존성이 적절한지 판단
- **개발자 (Developer)**: 실제 구현의 기술적 타당성을 검토한다
  - 코드 품질과 PEP 8 준수 확인
  - snake_case 네이밍 규칙 준수 확인
  - 한글 주석이 빠짐없이 작성되었는지 확인
  - 성능과 보안 이슈 점검
- **검수자 (Reviewer)**: 완성된 코드의 품질을 최종 검증한다
  - 테스트 코드가 충분한지 확인
  - 엣지 케이스 누락 여부 점검
  - 로그 기록이 `logs/` 폴더에 올바르게 설정되었는지 확인

## 3단계: Action (실행)

> 검토가 완료되면 실제 파일을 생성하고 코드를 작성한다.

- 승인된 계획에 따라 파일 생성 및 코드 작성
- 각 모듈의 `tests/` 폴더에 테스트 코드 함께 작성
- `pytest`로 테스트를 실행하여 통과 확인
- 완료 후 변경 사항을 커밋

---

## 워크플로우 요약

```
[Plan] 계획 수립 → 사용자 컨펌
  ↓
[Agent Team] 편집장 → 개발자 → 검수자 다각도 검토
  ↓
[Action] 코드 작성 → 테스트 → 커밋
```
